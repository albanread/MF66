This confirms the critical GC problem. `RootScanner::visit(&mut Word)` **reads `*slot` and classifies it via `HeapLayout::classify(raw: u64)`** — the GC decides whether a slot is a pointer by inspecting tag bits in the word itself. That is fundamentally incompatible with raw Forth data-stack cells, which carry no tags. I now have enough to write the review.

# MF66 Apple Silicon Port Plan — Adversarial Review

Prioritized, most load-bearing first. Verified against the actual substrate (`JASM/rust/src/a64/`, `native_macos.rs`, `newgc-core/src/`).

## 1. CRITICAL — The GC root-scanning model does not fit Forth's untagged data stack (the plan's "small / ~30–50 lines" estimate is the single biggest under-estimate)

`RootScanner::visit(&mut Word)` (`scanner.rs:39`) reads `*slot` and delegates to `PageEvacuator::visit`, which calls `HeapLayout::classify(raw: u64) -> WordKind` (`lisp_layout.rs:43`) to decide whether the slot is a pointer worth evacuating. **The GC infers pointer-ness from tag bits in the word.** This is the Lisp model: every root slot is a tagged `Word`.

Forth data-stack cells are **raw 64-bit integers with no tags**. The number `0x011D_2A40` on the stack is indistinguishable from a heap pointer with the same bit pattern. The plan's §2.5 hand-waves this — "**data stack** (TOS in x0 *plus* spilled cells from DSP top down to SP0) ... Each root visited via `RootScanner::visit(&mut Word)`" — but you **cannot** feed raw Forth cells to a classify-based evacuator:
- If you visit every data-stack cell as a `Word`, an integer that happens to look like a `String`-tagged pointer (tag=100) gets "evacuated" → heap corruption, the exact failure the plan's own risk section warns about but mis-attributes to "missing a root."
- Conversely the GC is a **moving/evacuating** collector (`PageEvacuator`, `cycle.rs`). Moving objects means it *rewrites the slot*. You cannot move an object that an untagged integer-or-pointer cell might reference, because you cannot safely update an ambiguous root — this is the classic conservative-vs-precise-GC wall. A precise moving GC requires precise roots; Forth's data stack is not precise.

The real answer (which the kernel review actually implies and the plan ignores): **the GC only manages a tagged sub-heap** — `FloatVec/RefVec/String/Builder` objects reached through the `RefVec` graph and the `HEAPPTR`/`LITERAL` regions (`runtime.rs:634-640`). Those regions are tagged. The raw data/return stacks are **not** GC roots in the precise sense; managed-object handles only enter the data stack as tagged values, and the kernel's 3-bit tag scheme (FloatVec=010, RefVec=011, String=100, Builder=101) is what makes a stack cell classifiable. So:
- The root set is the tagged regions + any **tagged** cells on the stacks — and the layout must classify untagged fixnum-like cells as `WordKind::Immediate` (non-pointer) by the *same* tag discipline Lisp uses for fixnums.
- This forces a hard invariant: **every managed handle on the data stack must always carry its tag** (no "raw pointer in a cell" ever, even transiently inside a primitive), or precise GC is unsound. WF66's x86 kernel may currently keep raw untagged pointers in cells in places; that has to be audited kernel-wide. This is not 30–50 lines; it is a correctness property spanning every primitive that touches a managed object.

**Correction:** Re-scope §2.5 from "small, ~30–50 lines, defer to MacNCL" to a first-class subsystem with its own design section: (a) state explicitly that GC is precise over the *tagged managed heap only*, not the raw stacks; (b) specify the tag invariant for stack cells and prove `classify` returns `Immediate` for every untagged value; (c) audit every kernel primitive that can hold a managed pointer untagged across a potential collection point; (d) decide what happens to a managed object referenced *only* from a raw integer cell (answer: it must never happen — enforce by construction). The "lift `Wf64Layout` directly, point at newgc-core" claim is fine for the *layout* but does nothing for the *root precision* problem, which is the actual hard part.

## 2. CRITICAL — x16 collides with the JIT loader's far-call veneers; the "x16 is free at call sites" claim is false on this substrate

The plan picks **x16 (IP0) as the indirect-call target / `CallTmp`** (§2.1 indirect call, §2.2 "Call scratch") and asserts it is "guaranteed free at call sites." But the *actual* `MacJit` you depend on builds every out-of-±128 MB call through an **absolute veneer that clobbers x16**: `abs_veneer` emits `movz/movk x16 …; br x16` (`native_macos.rs:60-67`, `:200-213`). So:
- The "STC uses `bl rel26` … `movz/movk + blr` the far fallback" path is wrong about *who* does the far jump. The loader, not your `render()`, materializes far branches — and it does so by overwriting x16 *at the call site* during relocation. If your code put a live value in x16 (e.g., the call target it computed, or anything it expects to survive the `bl`), the veneer destroys it.
- AAPCS64 says x16/x17 may be clobbered by *any* call (the linker/loader is permitted to insert veneers that use them). That is exactly what happens here. So x16 is correct as a *throwaway* scratch *immediately before* `blr`, but you must treat x16/x17 as **dead across every `bl`/`blr`**, including the implicit veneers the loader inserts. The plan's table marks x16 caller-saved (good) but the prose "guaranteed free at call sites" invites someone to keep something in it.

**Correction:** (a) Document that the `MacJit` veneer path owns x16; never assume a value in x16 survives any control transfer. (b) x17 is *also* not merely "leave for linker veneers" — confirm the loader doesn't use x17 too (it currently uses only x16, but pin x17 as reserved-for-loader anyway, do not put it in any pool). (c) The fusion-pool and pin-pool selection (x9–x15) is fine, but spell out that x16/x17 must never appear in any allocatable pool.

## 3. HIGH — x18 is not enforced anywhere in the toolchain; the encoder will silently emit it

The plan correctly says x18 is the Darwin platform register and "never touch." But the A64 encoder (`encode.rs`) has **no x18 reservation** — `grep` shows it happily encodes any `xN`. So the *only* thing standing between you and a corrupted platform register is discipline in `RegFile`/pool lists. The hand-written kernel (13.6k lines of new AArch64) is the real risk: a single scratch use of x18 in any `.masm` primitive is a silent, intermittent crash (Darwin uses x18 for thread-local/ABI purposes and the kernel may clobber it between your call-outs).

**Correction:** Add a mechanical guard — a lint/grep gate in CI that fails if `x18`/`w18` appears in any generated `render()` output *or* any kernel `.masm` file, and assert x18 is absent from every pool in `RegFile`. Don't rely on reviewer vigilance across 13.6k lines.

## 4. HIGH — The deferred-assembly retarget is under-scoped on three x86-shaped assumptions, not just `CellAlu`

The plan treats the optimizer passes as "free / register-name-agnostic once `Instr` carries abstract names." Three problems it under-weights:

**(a) Variable-length → fixed-width breaks `promote`/`window_fuse` cost models, not just the kernel peephole.** The plan correctly defers the *kernel* peephole (K4) for fixed-size assumptions, but the *Rust-side* passes also bake x86 instruction economics. `coalesce_dsp` is genuinely arch-neutral (offset arithmetic). But `promote_hot_cells` and `window_fuse` make promote/fuse decisions on the premise that a folded `op reg,[rbp+disp]` is *one* instruction. On AArch64 that's `ldr`+`op` = two fixed 4-byte instructions, and a promoted cell that saved one x86 load may now cost an extra `ldr` per use plus a scratch. The break-even point for promotion/fusion is **different**, so the passes can pessimize (emit more code than the naive path) while still being "correct." The plan acknowledges the `CellAlu` quality regression but not that it **invalidates the promote/fuse heuristics' thresholds**. These need re-tuning, not just a wider pool — and "measure in Phase 9" is too late; a pessimizing optimizer that ships in the Phase-5 baseline is worse than no optimizer.

**(b) Flags-as-data.** x86 `sbb rax,rax` and the carry-smear idioms encode flag *state* into a value across instructions. The plan handles the *comparison-word* cases (`csetm`), but `window_fuse`/reduction may move or reorder instructions assuming flag liveness is irrelevant (x86 flags are implicit and the passes never modeled them). On AArch64 NZCV is set by `cmp`/`fcmp`/`adds` etc. and consumed by `b.cc`/`csel`/`fcsel`. If any pass reorders a flag-setter past a flag-consumer (legal when "flags" were just an x86 micro-detail the IR didn't track), the result is wrong. The plan's `Instr` AST (`AdjustDsp/LoadCell/StoreCell/RegMove/CellAlu`) has **no representation of NZCV liveness** — it never needed one on x86 because the fused windows were Raw-free arithmetic. Verify that no optimizer pass can separate a compare from its branch/select; the compare→branch fusion in `reduce` helps but the *post-lowering* AST passes are the risk.

**(c) The text→AST round-trip decision is right but the migration risk is under-stated.** Making `lower()` emit `Instr` directly (not text) is the correct call. But the plan keeps `parse_instrs` "for the x86 round-trip test and for ingesting `Raw` spans." `Raw` spans are the escape hatch for anything the optimizer doesn't model — and on AArch64 those Raw spans are now **AArch64 text the AArch64 `parse_instrs` must still recognize enough to not mis-coalesce around.** The plan says coalesce/fuse treat Raw as barriers, which is sound, but "retain parse_instrs only for x86 round-trip" contradicts needing it to delimit AArch64 Raw spans. Clarify: either Raw spans are opaque blobs (then no AArch64 parse needed, but you lose any fusion across them — fine), or you need a minimal AArch64 recognizer (more work than "small").

## 5. HIGH — Differential oracle: the committed-corpus story is sound, but "byte-identical observable state" is unachievable and the level-2 cross-check is weaker than claimed

The plan's claim "No x86 box needed at MF66 build time; the corpus is the oracle" is the right architecture and **is** achievable on a Mac. Two corrections:

- **"byte-identical observable state" overstates it.** The plan itself later concedes FP must match "within IEEE-754 identity" (Phase 4) and that machine bytes won't match. But it also lists "(d) the byte-for-byte dictionary/HERE region after forget normalization" as a diff target. The dictionary region contains **compiled machine code** (STC, code *is* in the dictionary) and **baked absolute addresses** (variable body addresses, xt pointers). Those are different bytes (different ISA, different mmap addresses) even after ASLR normalization. You cannot byte-compare the dict region across architectures. Restrict the dict diff to the **arch-neutral header/data fields** (link, name, flags, data cells), explicitly excluding the code body and any baked code address. The plan's "variable-stub *shape* is reusable, only embedded instructions change" (§2.3) directly contradicts the "byte-for-byte dict region" diff goal.

- **Level-2 (reducer cross-check) is genuinely free and strong — but it only covers the front-end.** The plan correctly notes this isolates reducer bugs from lowering/kernel bugs. Good. But the committed reduced-token stream must be captured from WF66 *on x86* once; if the reducer is later changed, the golden is stale. This is fine, just flag that level-2's golden has the same "frozen on x86" provenance as level-1 — it is not independently regenerable on the Mac without re-running WF66.

- **The Wine path (level 3) is the weakest link and the plan should drop it from the critical path explicitly.** WF66 is a JIT that emits x86 into `VirtualAlloc2` memory with a VEH crash handler; running it under Wine means Wine must correctly emulate `VirtualAlloc2 + MEM_ADDRESS_REQUIREMENTS`, W^X, and vectored exception handling for self-modifying/JIT code. That is exactly the class of thing Wine handles unreliably. Open question #3 ("is the Wine box maintained long-term") should be answered **no** in the design: freeze the corpus as the oracle after Phase 5, and expand it by running WF66 on a real x86 Windows box when needed (not Wine). Don't design CI around Wine.

## 6. MEDIUM — Frame-pointer (x29) requirement and the STC "RSP = return stack" model conflict on Darwin

The plan maps "Return stack → sp" and "x29/x30 (FP/LR — STC frame + return address)." Two Darwin-specific issues:

- **Apple platforms require a valid frame-pointer chain (x29) for *all* code that can be sampled/unwound** (it's an Apple ABI requirement, enforced by tooling and assumed by the kernel's sampling profiler and crash reporter). But WF66's STC model uses the *native call/ret stack as the Forth return stack* — every Forth `>r`/`r>`/nest/unnest manipulates the same stack the C ABI uses for return addresses, and x29 is **not** maintained as a frame pointer through Forth words (there are no C frames per Forth word). So when your Mach exception handler (Phase 7) or the OS crash reporter walks x29, it will see a garbage/absent chain through JIT'd Forth code. The plan's Phase-7 "read AArch64 `__darwin_arm_thread_state64` frame" only gets the faulting frame; **backtracing into JIT'd Forth is impossible via x29 unwinding** — you must walk the Forth return stack manually using your own code-range registry. The plan implies this ("keep code-range + Forth-dump registration") but doesn't call out that x29 is *unusable* as a unwind anchor in STC code, which also means: do not try to use x29 for anything frame-like; it's just another free callee-saved register here (or must be parked).

- **16-byte sp alignment at every call.** STC pushes/pops single 8-byte cells onto sp as the return stack. AAPCS64 requires sp 16-byte aligned *at the instant of any `bl`/`blr`*. A return stack at odd cell-parity when you call out to an rt_* function violates this and faults on Darwin (which enforces sp alignment more strictly than Linux in some paths). The kernel review flags this; the plan's `aapcs_call` macro must *actively re-align sp* (save the unaligned sp, align down, call, restore) on every call-out, because the Forth return stack will frequently be at 8-mod-16. This is per-call-site overhead the plan doesn't budget.

## 7. MEDIUM — W^X (`pthread_jit_write_protect_np`) is per-thread; this breaks both the agent scheduler AND any GC that runs on a different thread

The plan notes the per-thread W^X issue only for agents (Phase 9). But it's broader:
- `pthread_jit_write_protect_np` toggles the calling thread's view of the MAP_JIT region between RW and RX. **The kernel compiles new definitions at runtime** (`:` … `;` writes machine code into the JIT arena) — every compile must toggle to RW, write, toggle to RX, on the *compiling thread*. If the IDE (Phase 8) runs `eval` on the AppKit main thread while anything else touches the JIT region, the protection state is per-thread and they will not agree.
- The MacNCL GC is described as "multi-mutator." If a collection or the IDE ever runs on a thread other than the one that holds the JIT region RW/RX, you get either a write fault (RX when you need RW) or you compile into an RW page another thread sees as RX. The plan's "single-threaded W^X means agents must share one mutator thread" is the right instinct but should be promoted to a **whole-runtime invariant**: MF66 is single-threaded for JIT-write purposes; the IDE eval must marshal to the one mutator thread, not call `eval` directly on the AppKit thread. Note this also reintroduces the plan's own Phase-8 latency risk (a long Forth loop freezes the window) — acknowledged in the MacNCL review but not connected to the W^X constraint.

## 8. MEDIUM — FTOS = d8 is correct, but the "only low 64 bits preserved" detail hides a real hazard

The plan is right that **d8, not d31**, is the FTOS home (d8–d15 low-64 are callee-saved; d31 is caller-saved and dies across libm). Good catch, and it directly contradicts four of the six reviews which all said d31 — the plan correctly overrides them. But:
- AAPCS64 preserves **only the low 64 bits of v8–v15**. If any kernel/emit path ever uses the *full* 128-bit `q8` (e.g., a SIMD `cnt`/vector op that happens to allocate v8, or a future vectorized float-vec primitive), the upper 64 bits are clobbered by callees and FTOS-as-q8 corrupts. The plan's `cnt` popcount and any NEON use must avoid v8–v15 for vector work, or restrict to the low-64 lane. Add this to the d8 grep-audit: not just "no residual d31," but "no 128-bit q8–q15 use."
- The plan keeps the fusion pool at x9–x15 (7 GP regs) but doesn't define an **FP scratch pool discipline** beyond "d0–d7." For FP-heavy fused windows with FSP coalescing (FSP in x22), the FP temps come from d0–d7 (caller-saved, fine window-local) — but any FP value that must survive a libm call inside a window needs a callee-saved d9–d15 slot, mirroring the GP promotion pool. The plan has a GP promotion pool (x23/x24) but **no FP promotion pool**. FP-heavy code (the Phase-4 Mandelbrot/FFT target) will want one.

## 9. MEDIUM — Kernel rewrite ordering is *not* cleanly buildable before all files are ported; the dependency claim is optimistic

"Phases are cumulative, dependency order strict left-to-right" and "you can boot before all files are ported" is partially true but glosses over hard cycles:
- The macro library (K0) genuinely gates everything — correct.
- But Phase 2 claims you can "boot the kernel headless" with K1 (mechanical files) *before* K2 (arith/compare/strings). The **interpreter/QUIT loop and `number.masm` depend on `compare` and `arith`**: parsing a number does base multiply/divide (needs `arith` div), and the interpreter's STATE/`find` dispatch does comparisons. You cannot run "integer arithmetic, stack ops, variables" (Phase 2 verify) without at least the comparison primitives and the multiply/divide used by number parsing. The plan puts `number.masm` in K1 (Phase 2) but its 128-bit-ish base conversion uses the same `cqo+idiv`→`sdiv`/`udiv` work that's scheduled in K2 (Phase 3). So the K1/K2 split is not a clean cut — `number.masm` straddles it.
- `dict.masm` (in K1/Phase 2) does hash compares; the plan itself flags the 32-bit `cmp eax,imm32` upper-half audit. `find` is on the boot-critical path, so that audit is a Phase-2 blocker, not a nice-to-have.

**Correction:** Re-draw the K1/K2 boundary around *boot-criticality*, not *mechanical-vs-hard*. Minimum bootable kernel = macros + stack + rstack + memory + the *subset* of arith (just `+ - * /mod`), the *subset* of compare (`= < 0= 0<`), dict find, number, parse, interpreter. The "mechanical first" ordering is a false economy if it leaves you unable to boot until a "hard" file lands anyway.

## 10. LOW/MEDIUM — Taint-set re-validation is correctly flagged but the LR/x30 hazard is bigger than stated

The plan flags return-stack interference because "LR/x30 is special." Concretely: in STC, `>r`/`r>` move values between the data stack and the **native return stack (sp)**, and `bl` writes x30 (LR) which then gets pushed by the callee's prologue. On x86 the return address is pushed by `call` automatically and `>r` just manipulates `[rsp]`. On AArch64, **the return address lives in x30 until explicitly stored**, and a leaf Forth word may never spill it. So `r@`/`r>` semantics ("top of return stack") must decide: is the "top of the return stack" the value in x30, or the value at `[sp]`? This is a genuine semantic divergence the taint set never had to model on x86 (where it's unambiguously `[rsp]`). The plan's "re-audit each taint rule" is right but should specifically call out that **`>r`/`r>`/`r@`/`2>r`/`rdrop` need a from-scratch AArch64 semantic definition**, not a translation — they are not in the "mechanical" bucket the kernel review put them in.

## 11. LOW — Position-independence / ADRP discipline is stated correctly but the loader already resolves `@PAGE/@PAGEOFF`, so "default to movz/movk" may be leaving performance on the table for no PIC benefit

The plan insists `render()` default to movz/movk absolutes "to preserve the same PIC discipline x86 had with movabs." But the JIT arena is `mmap`'d at a runtime address and the loader patches relocations *after* layout (`native_macos.rs` applies relocations including the adrp page-offset path, `:224`). Since the loader resolves addresses post-placement anyway, movz/movk absolutes are *also* only correct because the loader bakes them — there's no PIC advantage to movz/movk over `adrp+add` here; both are loader-resolved absolutes. The real distinction is **reach and size** (`adrp+add` is 2 insns / ±4GB; movz/movk is 4 insns / full 64-bit), not PIC. The plan's framing ("movz/movk preserves PIC, adrp breaks it") is incorrect for this loader. Recommend: prefer `adrp+add` for data within ±4GB of the arena (smaller, faster), fall back to movz/movk only when out of range — the loader supports the fixups (verified: `@PAGE/@PAGEOFF` in encoder, adrp patch in loader). This *inverts* the plan's default.

## 12. LOW — Effort accounting: "13.6k lines, file-by-file mechanical" undersells the verification cost, and the per-file "effort: small" labels are misleading

The line-count is the *translation* cost. The dominant cost in a hand-written assembly kernel port is **per-primitive differential verification** (the kernel review says "test each primitive in isolation" and budgets a "1–2 week debugging cycle" just for ABI). With ~30 files and the noted silent-corruption failure modes (`rep movs` expansion, flag idioms, 128-bit mul/div high-half, locals displacement overflow), the realistic effort is dominated by the test matrix, not the translation. The plan's phase gates ("full integer ANS core suite passes") are the right verification structure, but the per-row "effort: small/medium" labels in §2.3/§2.4 measure typing, not getting-it-right. Recommend re-labeling effort as (translate / verify) pairs, because the verify column is where the schedule actually lives — and it's serialized behind the differential corpus being complete.

### Net assessment
The plan's **strongest** correct calls: TOS=x0, the callee-saved DSP/UP/LP/FTOS=d8 mapping (correctly overriding the reviews' d31 error), upgrading LP and the promotion pool to callee-saved, the `lower()→Instr`-AST refactor, and the committed-corpus oracle as the Mac-local workhorse. Those are sound under AAPCS64.

The plan's **most dangerous** gaps, in order: (1) the precise-moving-GC vs untagged-Forth-stack impedance mismatch is mis-scoped as trivial and is actually a kernel-wide invariant; (2) x16 is contended with the loader's veneers, falsifying "guaranteed free"; (3) x18 has zero toolchain enforcement across 13.6k hand-written lines; (4) the optimizer's promote/fuse *thresholds* (not just its register names) are invalidated by fixed-width + no-memory-ALU and risk pessimizing the Phase-5 baseline; plus the Darwin-specific sp-alignment-per-call and per-thread-W^X invariants that the plan scatters across "Phase 9" rather than treating as runtime-wide.
